# Learn Break - 壊して学ぶ

「あえて壊してみる」ことで、コードの動作原理を深く理解するスキルです。
Anthropicの研究によると、デバッグ能力はAI依存で最も差が出る能力です。

## 目的

> 一部の数値を書き換えたり、条件を外したりして、どう挙動が変わるか、どんなエラーが出るかを確認する

意図的に「痛みを伴う試行錯誤（Cognitive effort）」を自分に課すことで、デバッグ能力を鍛えます。

## 使用方法

```bash
# 現在のブランチの変更について壊すシナリオを提案
/learn-break

# IDEで選択中のコードについて
/learn-break --selection

# 特定ファイルについて
/learn-break --file apps/src/utils/validation.ts

# フォーカスエリアを指定
/learn-break error        # エラーハンドリング系
/learn-break edge         # エッジケース・境界値
/learn-break performance  # パフォーマンス・無限ループ

# 組み合わせ
/learn-break --selection error
```

## 実行フロー

### 1. 対象コードの特定

$ARGUMENTSを解析：

- `--selection` → IDEで選択中のコードを対象
- `--file <path>` → 指定ファイルを対象
- 空 / その他 → 現在のブランチの変更（develop比較）を対象

フォーカス指定（`error`, `edge`, `performance`）があれば適用

### 2. 脆弱ポイントの分析

対象コードを分析し、「壊せるポイント」を特定：

- **条件分岐**: if文の条件を反転/削除したら？
- **境界値**: 配列の最初/最後、0、null、空文字
- **非同期処理**: awaitを外したら？順序を変えたら？
- **エラーハンドリング**: try-catchを外したら？
- **型**: 型を無視したら？anyを使ったら？
- **依存関係**: 引数を変えたら？undefinedを渡したら？

### 3. 壊すシナリオの提案

**提案数**: 3〜5シナリオ

各シナリオで以下を提示：
1. **変更内容**: 具体的に何をどう変えるか
2. **予想課題**: 何が起きると思うか考えてもらう
3. **実験方法**: どうやって確認するか
4. **学習ポイント**: この実験で何がわかるか

### 4. 実験サポート

ユーザーが実験を行う際のサポート：
- 予想が当たったか確認
- 予想外の挙動があれば解説
- デバッグのヒント（答えは教えない）

## 出力フォーマット

### シナリオ提案

```markdown
## 🔧 壊して学ぶシナリオ

対象: `path/to/file.ts` の `functionName` 関数
フォーカス: エラーハンドリング

---

### 🧪 シナリオ1: 早期リターンを削除

**変更内容**:
```typescript
// Before
if (!user) return null;

// After (この行を削除またはコメントアウト)
// if (!user) return null;
```

**実験前に予想してください**:
- 何が起きると思いますか？
- どんなエラーが出ると思いますか？
- どの行でエラーになると思いますか？

**確認方法**:
1. 上記の変更を加える
2. `{{DEV_COMMAND}}` で起動
3. ユーザー未ログイン状態でページにアクセス

**学習ポイント**:
早期リターン（ガード節）がなぜ重要か体感する

---

### 🧪 シナリオ2: awaitを外す

**変更内容**:
```typescript
// Before
const data = await fetchData();

// After
const data = fetchData(); // awaitを削除
```

**実験前に予想してください**:
- `data` にはどんな値が入りますか？
- 後続の処理はどうなりますか？
- エラーは出ますか？出るとしたらどこで？

**確認方法**:
1. 変更を加える
2. console.log(data) を追加して値を確認
3. 後続の処理の挙動を観察

**学習ポイント**:
Promiseの仕組み、async/awaitの動作原理

---

### 🧪 シナリオ3: 空配列を渡す

**変更内容**:
```typescript
// テストコードまたはコンソールで以下を実行
processItems([]);  // 空配列を渡す
```

**実験前に予想してください**:
- 何が返ってきますか？
- エラーは出ますか？
- 境界値チェックはされていますか？

**学習ポイント**:
エッジケースの処理、防御的プログラミング

---

## 💡 実験のコツ

1. **変更前に予想を立てる**: これが最も重要！
2. **一つずつ変える**: 複数同時に変えると原因がわからなくなる
3. **元に戻せるように**: git stash や別ブランチで実験
4. **エラーメッセージを読む**: スタックトレースから原因を追う

## 🚨 注意

- 実験後は必ず元に戻してください: `git checkout -- <file>`
- 本番環境では絶対に実験しないでください
- テストコードで実験するのがベストです

---

実験する準備ができたら、予想を教えてください！
シナリオ番号と、あなたの予想を入力してください。
```

### 実験後のフィードバック

```markdown
## 📊 実験結果

### シナリオ1: 早期リターンを削除

**あなたの予想**: nullアクセスでエラーになる
**実際の結果**: TypeError: Cannot read property 'name' of null

✅ 予想通り！

**解説**:
- エラー発生箇所: L25 `user.name`
- `user` が `null` の状態でプロパティアクセスしたため
- 早期リターンはこの種のエラーを防ぐガード節として機能

**さらに深掘り**:
- TypeScriptの `strictNullChecks` を有効にすると、コンパイル時にこのエラーを検出できます
- オプショナルチェーン `user?.name` でも防げますが、意味が異なります

---

### 🎯 学んだこと

1. ガード節の重要性
2. null/undefinedチェックの必要性
3. TypeScriptの型チェックが防いでくれること

💡 `/learn-quiz` で理解度をテストしてみましょう
```

## フォーカスエリア別のシナリオ例

### error（エラーハンドリング）
- try-catchを外す
- エラーを握りつぶす（空のcatch）
- 非同期エラーのハンドリングを外す

### edge（エッジケース）
- 空配列、空文字、null、undefined
- 配列の最初/最後の要素
- 0、負の数、非常に大きな数
- 特殊文字、Unicode

### performance（パフォーマンス）
- 無限ループの条件
- N+1クエリ
- 不要な再レンダリング
- メモ化を外す

## デバッグ支援モード

実験中にエラーが出て困ったら「ヒントをください」と言ってください。

**ヒントの段階**:
1. **方向性ヒント**: どの辺りを見るべきか
2. **具体的ヒント**: エラーの種類、関連する概念
3. **解決策**: 最終手段として答えを提示

> 「答え」を聞くのではなく、「原因の切り分け方」や「デバッグの戦略」を相談する
