# Learn Quiz - 理解度クイズ

AIが生成したコードや変更内容について、理解度を試すクイズを出題します。
「動いたから終わり」ではなく、本当に理解しているかを確認するためのスキルです。

## 目的

> 「今書いてもらったコードの内容について、私の理解度を試すクイズを出して」

受動的にコードを受け取るだけでなく、能動的に理解を確認することで学習効果を高めます。

## 使用方法

```bash
# 現在のブランチの変更についてクイズ（デフォルト）
/learn-quiz

# IDEで選択中のコードについてクイズ
/learn-quiz --selection

# 特定ファイルについてクイズ
/learn-quiz --file apps/src/hooks/useAuth.ts

# 難易度を指定
/learn-quiz easy      # 基本的な理解度
/learn-quiz medium    # 応用レベル
/learn-quiz hard      # 深い理解・エッジケース

# 組み合わせ
/learn-quiz --selection hard
```

## 実行フロー

### 1. 対象コードの特定

$ARGUMENTSを解析：

- `--selection` → IDEで選択中のコードを対象
- `--file <path>` → 指定ファイルを対象
- 空 / その他 → 現在のブランチの変更（develop比較）を対象

難易度指定（`easy`, `medium`, `hard`）があれば適用、なければ `medium`

### 2. コード分析

対象コードを分析し、以下の観点でクイズポイントを抽出：

- **処理フロー**: 何がどの順序で実行されるか
- **データ変換**: 入力から出力への変換ロジック
- **条件分岐**: どの条件でどの処理が走るか
- **エラーハンドリング**: 例外・エラーの扱い
- **設計判断**: なぜその実装を選んだか

### 3. クイズ出題

**出題数**: 3〜5問（難易度とコード量に応じて調整）

#### 難易度別の出題傾向

| 難易度 | 出題傾向 |
|-------|---------|
| easy | 処理の流れ、基本的な動作確認 |
| medium | 条件分岐、エラーケース、型の理解 |
| hard | エッジケース、代替実装、設計判断の理由 |

### 4. 回答待ち & フィードバック

ユーザーの回答を待ち、以下を提供：

- 正解/不正解の判定
- 正解の解説
- 関連する補足知識

## 出力フォーマット

### クイズ出題

```markdown
## 🧠 理解度クイズ

対象: `path/to/file.ts` の変更内容
難易度: ⭐⭐ (medium)

---

### Q1. 処理フローについて

この関数が呼ばれたとき、最初に実行される処理は何ですか？

A) データベースへのクエリ
B) 入力のバリデーション
C) キャッシュの確認
D) ログの出力

---

### Q2. エラーハンドリングについて

`fetchUser` が失敗した場合、どうなりますか？

A) nullを返す
B) 例外がそのままthrowされる
C) デフォルト値を返す
D) リトライが実行される

---

### Q3. 設計判断について（記述式）

なぜここで `useMemo` ではなく `useCallback` を使用していますか？
理由を説明してください。

---

💡 回答を入力してください（例: Q1: B, Q2: A, Q3: ○○だから）
```

### フィードバック

```markdown
## 📊 クイズ結果

### スコア: 2/3 (67%)

---

### Q1. ✅ 正解！
あなたの回答: B) 入力のバリデーション
→ 正解です。早期リターンパターンで、不正な入力を最初にチェックしています。

### Q2. ❌ 不正解
あなたの回答: A) nullを返す
正解: B) 例外がそのままthrowされる

**解説**: この関数では `fetchUser` の例外をキャッチしていません。
呼び出し元でtry-catchする設計になっています。
コードの該当箇所: L15-20

### Q3. ⭕ 部分点
あなたの回答: 関数を返すから
→ 方向性は正しいです！

**補足解説**:
`useCallback` は「関数自体」をメモ化します。
この場合、`onClick` ハンドラを子コンポーネントに渡すため、
関数の参照が変わらないようにして不要な再レンダリングを防いでいます。

---

## 🎯 復習ポイント

1. **エラーハンドリング**: 例外がどこでキャッチされるか確認する習慣をつけましょう
2. **Hooksの使い分け**: useMemo vs useCallback の違いを復習

💡 `/learn-explain` でコードの詳細解説を見ることもできます
```

## クイズの種類

### 選択式（多肢選択）
- 処理フロー、条件分岐、エラーケースの確認に使用
- 明確な正解がある問題

### 記述式
- 設計判断の理由、代替案との比較に使用
- 理解の深さを確認する問題

### コード補完式
- 一部を隠して穴埋め
- 実際に書けるかの確認

```markdown
### Q4. コード補完

以下のコードの `???` に入るものは？

```typescript
const result = data.filter(item => ???);
```
```

## 難易度別の出題例

### Easy（基礎理解）

```
Q: 次のコードは何をしていますか？

const user = await fetchUser(id);
console.log(user.name);

A) ユーザーIDから非同期でユーザー情報を取得し、名前をログ出力
B) ユーザーの名前を同期的に取得する
C) ユーザーIDをログ出力する
```

### Medium（応用理解）

```
Q: 以下のコードで、error が undefined のときの挙動は？

try {
  await riskyOperation();
} catch (error) {
  console.log(error.message);
}

A) エラーメッセージを出力
B) 'undefined.message' を出力
C) エラーが発生する可能性がある
```

### Hard（深い理解）

```
Q: このコンポーネントが過剰に再レンダリングされる原因は何か、
またどう修正すべきかを説明してください。

コード例を記述
```

## 学習効果を高めるコツ

1. **正直に答える**: わからないときは「わからない」と答える
2. **解説を読む**: 不正解の問題は特に解説をしっかり読む
3. **再チャレンジ**: 時間を置いて同じコードで再度クイズを受ける
4. **深掘り**: 気になった点は `/learn-explain` で詳しく聞く

## クイズの使用タイミング

1. **実装直後**: コードの理解度を確認
2. **レビュー後**: 指摘された点を理解したか確認
3. **時間経過後**: 記憶の定着度を確認

## スコア解釈ガイド

| スコア | 判定 | 次のアクション |
|-------|------|----------------|
| 90%以上 | 完全理解 | 他のコードに応用可能 |
| 70-89% | 部分理解 | `/learn-explain` で詳細確認 |
| 50-69% | 浅い理解 | `/learn-break` で実験的に学習 |
| 50%未満 | 不十分 | 一から学び直しが必要 |
