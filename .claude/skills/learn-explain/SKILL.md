# Learn Explain - 生成後理解プロセス

AIが生成したコードや変更内容を深く理解するためのスキルです。
Anthropicの研究に基づく「Generation-then-comprehension」アプローチを実践します。

## 目的

> AIが書いたコードを「借り物」ではなく、ロジックを1行ずつ説明できるレベルまで「自分の知識」に昇華させる

## 使用方法

```bash
# 現在のブランチの変更を解説（デフォルト）
/learn-explain

# IDEで選択中のコードを解説
/learn-explain --selection

# 特定ファイルを解説
/learn-explain --file apps/src/app/page.tsx

# 特定の観点で解説
/learn-explain "エラーハンドリングについて"
/learn-explain --selection "なぜこのパターンを使ったか"
```

## 実行フロー

### 1. 対象コードの特定

$ARGUMENTSを解析：

- `--selection` → IDEで選択中のコードを対象
- `--file <path>` → 指定ファイルを対象
- 空 / その他 → 現在のブランチの変更（develop比較）を対象

**diffモードの場合**:
```bash
git diff develop --name-only  # 変更ファイル一覧
git diff develop              # 詳細な変更内容
```

**選択コードがある場合**:
`$SELECTION` 変数から取得

### 2. コンテキスト収集

対象コードに関連する情報を収集：

- 変更されたファイルの全体像
- 関連する型定義やインターフェース
- 呼び出し元・呼び出し先のコード
- 関連するテストコード（あれば）

### 3. 段階的解説

以下の順序で解説を提供：

#### A. 概要レベル（What）
- このコードは何をしているか
- 全体的な処理フロー

#### B. 詳細レベル（How）
- 各関数・メソッドの役割
- データの流れ
- 状態の変化

#### C. 理由レベル（Why）- **最重要**
- なぜこのアプローチを選んだか
- 他の選択肢との比較
- トレードオフの説明

### 4. 深掘り質問

解説後、理解を深めるための質問を提示：

```
🤔 深掘りポイント:
1. 「なぜここで○○パターンを使ったか説明できますか？」
2. 「この処理を別のアプローチで書くとどうなりますか？」
3. 「エラーが発生した場合、どう伝播しますか？」
```

## 出力フォーマット

### 変更ファイルが複数ある場合

```markdown
## 📂 変更概要
- `path/to/file1.ts` - [簡潔な説明]
- `path/to/file2.ts` - [簡潔な説明]

## 📖 詳細解説

### path/to/file1.ts

#### What: 何をしているか
[説明]

#### How: どう実装しているか
[説明]

#### Why: なぜそうしたか
[説明]

---

### path/to/file2.ts
[同様の構造]

---

## 🤔 深掘りポイント
1. [質問1]
2. [質問2]
3. [質問3]

💡 気になる点があれば「○○について詳しく」と聞いてください
```

### 単一ファイル・選択コードの場合

より詳細な行単位の解説を提供：

```markdown
## 📖 コード解説

### 行ごとの解説

```typescript
// L1-5: 型定義
// → なぜこの型構造にしたか: ...

// L7-15: メイン関数
// → このアプローチを選んだ理由: ...
```

### 代替アプローチ

別の書き方との比較：

| アプローチ | メリット | デメリット |
|-----------|---------|-----------|
| 現在の実装 | ... | ... |
| 代替案A | ... | ... |

### 🤔 深掘りポイント
1. [質問]
```

## 学習効果を高めるコツ

1. **受動的に読まない**: 解説を読んだら、自分の言葉で要約してみる
2. **疑問を持つ**: 「なぜ？」と思った点は追加で質問する
3. **比較する**: 代替アプローチとの違いを理解する
4. **実践する**: 類似のコードを自分で書いてみる

## よくある質問パターン

### 「なぜこのライブラリを使ったのか」

```
背景: ○○というタスクがある
選択肢:
  1. 標準APIで実装（メリット: シンプル、デメリット: 手間がかかる）
  2. ライブラリA（メリット: 機能豊富、デメリット: バンドルサイズ大）
  3. ライブラリB（メリット: 軽量、デメリット: 機能制限）

選択理由: ライブラリBを選んだ → バンドルサイズを重視した
トレードオフ: 今後新機能が必要になったら検討の余地あり
```

### 「別の書き方はあるか」

```
現在の実装:
  [コード例]

代替案1: 関数型アプローチ
  メリット: イミュータブル、テストが容易
  デメリット: パフォーマンスオーバーヘッド

代替案2: クラスベース
  メリット: カプセル化、状態管理が容易
  デメリット: ボイラープレートコード増加
```
