# LLM入力支援拡張機能 仕様書

## プロダクト名

**Pathline**

### 名前の意図
- ユーザーの曖昧な入力から、意図の"道筋"を見つけて整える
- 入力途中で候補がぶれず、意図が収束していく体験と一致する
- LLM、ChatGPT、Claude など特定サービス名に依存しない
- 英語圏でも比較的発音しやすく、プロダクトの抽象度とも合う

### タグライン案
- Find the right path for your prompt.
- Turn rough input into clear requests.
- Guide your input into better prompts.

---

# 1. 背景

LLMを使うユーザーは増えているが、多くのユーザーは「何をどう頼めばよいか」を毎回考える必要がある。特に以下の課題がある。

- 雑に入力すると、期待した出力が返らない
- 毎回プロンプトを整えるのが面倒
- 要約、改善、質問整理、レビュー依頼などの意図を、明確な依頼文に変換する作業が発生する
- 入力の途中では意図がまだ曖昧だが、入力が進むにつれてユーザーのやりたいことは徐々に収束していく

既存のプロンプト最適化ツールは存在するが、以下の点で改善余地がある。

- プロンプトエンジニア向けで難しい
- UIが重い
- 入力フローを止める
- 候補が多すぎる、または都度変わりすぎる

Pathline は、ユーザーの入力中テキストをもとに、**安定した単一候補をインラインで提示し、必要に応じて軽く切り替えられる**ことを中核体験とする。

---

# 2. プロダクトビジョン

## 一文定義
Pathline は、**人の雑な入力を、LLM が理解しやすい依頼文へリアルタイムに収束させる入力支援拡張機能**である。

## 解く問題
- LLM にうまく頼めない
- 入力しながら意図を整理したい
- 速く、自然に、考えを止めずにプロンプト化したい

## 提供価値
- 入力の途中で、最適な依頼文の方向性が見える
- ユーザーは自分で整形しなくてもよい
- 速く、軽く、邪魔しない
- ChatGPT 特化ではなく、LLM 全般に使える入力支援レイヤーとして機能する

---

# 3. 対象ユーザー

## 一次ターゲット
LLM を高頻度に使う PC ユーザー

### 具体像
- エンジニア
- 企画職、PM、PdM
- ビジネス職で生成AIを日常的に使う人
- ライター、リサーチャー、ナレッジワーカー

## ペイン
- 入力が雑で、期待値通りの回答が出ない
- 毎回「どう頼めばいいか」を考えるのが面倒
- 長文を貼ってから、要約させるのか、整理させるのか、レビューさせるのかを頭の中で組み立てている

## なぜ拡張機能か
- 利用シーンがブラウザ上に集中している
- LLM サービス自体を置き換えるのではなく、入力フローに差し込む方が自然
- アプリよりも「その場」で価値を出しやすい

---

# 4. プロダクトコンセプト

## コンセプト
**Grammarly for LLM input**

ただし、文章そのものを改善するのではなく、**LLM に対する依頼文の方向性を整える**ことに特化する。

## 体験の核
- 候補は常に 1 つを基本とする
- 候補は入力に応じて更新される
- ただしカテゴリは安定重視で、簡単には切り替わらない
- ユーザーは Tab で即確定できる
- 必要なら矢印キーで他候補に切り替えられる

---

# 5. MVP スコープ

## MVP の目的
- この UX に価値があるかを最速で検証する
- 候補の安定感と速度に対してポジティブな反応が得られるかを見る
- 最初のカテゴリ設計が妥当かを確認する

## MVP に含むもの
- テキスト入力中にインライン候補を表示
- 候補は 1 つをデフォルト表示
- 候補カテゴリをルールベースで判定
- Tab で候補を確定、入力欄へ反映
- ↑ / ↓ で候補カテゴリを軽く切り替え
- Esc で候補を閉じる
- debounce ベースの軽量更新

## MVP に含まないもの
- ログイン
- クラウド同期
- API 接続
- LLM による候補生成
- 課金
- 高度な設定UI
- チーム共有

---

# 6. UX 原則

## 原則1: 速さが最優先
- 賢さより速さ
- 毎回再生成ではなく、リアルタイムっぽい体験を作る
- 目標は体感 100ms 以下

## 原則2: 候補は安定していること
- 毎回カテゴリが変わらない
- 一度決まった方向性は、明確な理由がない限り維持する
- 入力でカテゴリを絞り込むイメージ

## 原則3: ユーザーに選ばせすぎない
- 最初から複数候補を並べない
- 最も良い 1 候補をまず出す
- 他候補は軽い操作でアクセス可能にする

## 原則4: フローを止めない
- パネル中心にしない
- 視線移動を最小にする
- キーボード操作だけで完結できるようにする

- 以下の内容を箇条書きで整理してください。

## ユースケース5: コードや文章のレビュー依頼

- 設計文

候補:

# 9. 候補決定ロジック

## 基本方針
- 毎回ゼロから候補を作るのではなく、意図カテゴリのスコアを更新する
- 入力に応じてカテゴリが徐々に絞られる
- トップカテゴリは簡単には変更しない
- カテゴリが同じなら、文面のみを洗練する

## スコアリング対象
- 文字数
- 改行数
- 疑問符
- 明示キーワード
- 曖昧表現
- コードっぽさ
- 構造化を示唆する表現

## 代表ルール
### 文字数が長い
- text.length > 120 → Summarize +3, Structure +1

### 改行がある
- text includes "\n" → Structure +2

### 疑問符がある
- text includes "?" or "？" → Clarify +2

### 曖昧改善語
- いい感じ / 直して / 整えて / 改善 → Improve +3

### 明示キーワード
- 要約 → Summarize +5
- レビュー / 問題点 → Review +5
- 整理 / まとめ → Structure +4
- どう思う / どうすれば → Clarify +3

### コード記号
- { } ( ) ; = など → Review +3

## カテゴリ切替ルール
- 新カテゴリが現カテゴリより一定以上優勢な場合のみ切り替える
- 例: スコア差が 2 以上、または 2 回連続で優勢
- ただし明示キーワードがある場合は即切り替え可

## 文面更新ルール
- カテゴリが維持される場合は、候補文の表現のみ洗練してよい
- 例: Improve のまま「簡潔に」「ビジネス向けに」などを足す

# 10. 表示・操作仕様

## 表示
- 入力欄内または直下にインライン候補を表示
- 候補は淡いゴースト表示または補助表示
- デフォルトでは 1 候補のみ表示

## キーボード操作
- Tab: 候補を確定して反映
- ArrowDown / ArrowUp: 他カテゴリ候補へ切り替え
- Esc: 候補を閉じる

## 表示タイミング
- 入力停止後 120〜200ms で再判定
- Enter、句点、疑問符、明示キーワードなどは即更新対象

## 非表示条件
- テキスト長が短すぎる
- フォーカスが外れた
- ユーザーが Esc を押した
- 対象外入力欄である

---

# 11. 対象DOMと技術的前提

## 対応対象
MVP では以下を優先する。
1. textarea
2. contenteditable

## 対象範囲
- 一般的な Web テキスト入力欄
- 特定の LLM サービスに依存しない

## 技術前提
- Chrome Extension Manifest V3
- content script で入力要素を監視
- 軽量なローカル判定
- 外部APIなし
- local storage に最低限の設定保存が可能

---

# 12. 非機能要件

## パフォーマンス
- 候補表示までの体感遅延は 100ms 前後を目標
- 再描画は必要時のみ
- 同一候補なら再描画しない
- スコアリングは同期でも十分軽い処理量に収める

## 安定性
- 候補が短時間で頻繁に切り替わらない
- 明示的な意図が出るまでは、大分類カテゴリを維持する

## プライバシー
- MVP では入力内容を外部送信しない
- ローカル処理のみ
- 対象ドメインや対象入力欄は必要最低限の監視とする

## 拡張性
- 将来的に AI 候補生成を追加可能な構造にする
- カテゴリやテンプレを差し替え可能な構造にする
- サービス別最適化を後付けできるようにする

# 15. リスクと対策

## リスク1: 候補がぶれすぎる
### 対策
- しきい値切替
- 連続優勢ルール
- カテゴリ内微調整優先

## リスク2: 候補が遅い
### 対策
- ルールベースのみで開始
- debounce 更新
- 差分が小さい場合は再評価スキップ

## リスク3: 候補がズレる
### 対策
- 最初の 5 カテゴリを絞る
- 明示キーワードは強く反映
- 定性フィードバックを見て辞書調整

## リスク4: 対応DOMが不安定
### 対策
- 最初は textarea と主要 contenteditable に絞る
- サイト個別最適化は後回し

# 16. 下流工程への引き継ぎ方針

実装担当は以下の順で進める。

## 実装優先順位
1. 対象入力欄の検出
2. 入力イベント監視
3. スコアリングエンジン実装
4. 候補カテゴリ決定ロジック実装
5. インライン候補描画
6. Tab / Esc / Arrow 操作実装
7. 再描画最適化
8. 簡易設定の土台実装

## 実装分離の推奨
- dom layer
- scoring engine
- category state machine
- template generator
- inline renderer
- keyboard handler

下流工程では、特に scoring engine と state machine を疎結合にすること。

# 18. 一言まとめ

Pathline は、**入力されたテキストからユーザーの意図を推定し、その意図を LLM 向けの依頼文へ安定的に収束させるインライン入力支援拡張機能**である。
